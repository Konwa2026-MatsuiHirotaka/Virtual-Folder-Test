# MQL5フォルダ構成管理基盤 詳細設計書

## 1. 目的と結論

React画面からMQL5バックエンドのフォルダ構成を表示・変更し、移動後に集約includeファイルを自動再生成する管理基盤を定義する。

| 結論 | 採用理由 |
|---|---|
| 構成モデルを正規情報源にする | 差分・検証・復元が可能 |
| Reactから直接ファイル操作しない | パス逸脱、競合、部分移動を防止 |
| `ProjectIncludes.mqh`を自動生成する | 利用側ファイルの配置自由度を保ち、物理配置変更を一点に集約 |
| 移動は計画→検証→適用に分ける | 壊れた構成を公開しない |
| 最終的にMQL5をコンパイルする | ファイル存在だけではinclude解決を保証できない |

## 2. 前提

| 項目 | 方針 |
|---|---|
| 対象 | `MQL5`配下のInclude、Experts、Indicators等 |
| UI | React |
| 管理バックエンド | Node.jsを第一候補。ファイルシステムへのアクセスはここに限定 |
| 正規情報源 | `project-layout.json`等の構成モデル |
| 変更単位 | 1つの構成変更コマンドをトランザクションとして扱う |
| 復元 | Gitまたは変更前スナップショット |
| 非対象 | 任意のOSフォルダを無制限に操作する汎用ファイルマネージャー |

## 3. 推奨Module構成

| Module | Interfaceの責務 | Implementationの責務 |
|---|---|---|
| `ProjectCatalog` | 構成取得、論理名からノード解決 | モデル読込、インデックス作成 |
| `LayoutPlanner` | 移動要求から変更計画を返す | 移動元・移動先・影響・生成対象の算出 |
| `LayoutValidator` | 計画と構成を検証 | ルート逸脱、衝突、循環、禁止領域、依存欠落の検証 |
| `FileSystemAdapter` | 読む、作る、移動する、削除する | Node.js `fs`または`fs-extra`による実操作 |
| `IncludeResolver` | 論理依存を物理includeへ変換 | 集約includeの決定的生成 |
| `ProjectTransaction` | 構成変更を適用し結果を返す | スナップショット、移動、保存、生成、ロールバック |
| `CompilerAdapter` | コンパイル結果を返す | MetaEditor起動、ログ解析 |
| `FileWatcher` | 外部変更イベントを通知 | chokidar等で監視・イベント集約 |
| `AuditLog` | 変更履歴を記録・取得 | 操作者、計画、結果、復元情報を保存 |

外部UIに公開するseamは`ProjectOrganizer`だけにする。内部ModuleをReactへ露出させないことで、UIがファイルシステム詳細に依存しない深いModuleにする。

## 4. 構成モデル

```json
{
  "schemaVersion": 1,
  "projectRoot": "C:/.../MQL5",
  "publicInclude": "Include/nautilus/ProjectIncludes.mqh",
  "nodes": [
    {
      "id": "mtf-signal",
      "kind": "include",
      "logicalPath": "nautilus/chartsyncx/mtf/MtfSignal.mqh",
      "physicalPath": "Include/nautilus/mtf/MtfSignal.mqh",
      "movable": true,
      "exports": ["MtfSignal"]
    }
  ]
}
```

`physicalPath`だけを正規情報源にせず、論理名と依存関係も保持する。移動時は`physicalPath`を更新し、集約includeを再生成する。

## 5. 集約includeの設計

```mql5
// AUTO-GENERATED. DO NOT EDIT.
#property strict
#include "C:\\...\\Include\\nautilus\\barflow\\ClosedBar.mqh"
#include "C:\\...\\Include\\nautilus\\mtf\\MtfSignal.mqh"
```

| ルール | 内容 |
|---|---|
| 編集 | 手編集禁止。構成モデルから生成 |
| 生成 | 決定的順序で出力し、同じモデルから同じ内容を生成 |
| 検証 | 生成前に存在、拡張子、ルート内所属を確認 |
| 失敗時 | 新ファイルを公開せず、前回正常版を維持 |
| 可搬性 | 絶対パスはPC依存。移設時に再生成 |

これにより利用側ファイルの配置自由度とinclude解決の一元管理を両立する。ただし、MQL5コンパイラの絶対パス構文は最初に最小ファイルで実証する。制限がある場合も、同じ集約方式を`MQL5/Include`基準の論理パスで実装できる。

## 6. 変更処理フロー

| 段階 | 処理 | 永続変更 |
|---|---|---:|
| 1 | UIが論理ノードと移動先を送信 | なし |
| 2 | `ProjectCatalog`が実パスを解決 | なし |
| 3 | `LayoutPlanner`が計画を作成 | なし |
| 4 | ルート、衝突、依存、禁止領域を検証 | なし |
| 5 | 対象ファイルとモデルをスナップショット | バックアップ |
| 6 | ファイルを移動・改名 | あり |
| 7 | 構成モデルを保存 | あり |
| 8 | 集約includeを一時ファイルへ生成後置換 | あり |
| 9 | MetaEditorでコンパイル検証 | ログ |
| 10 | 成功を監査ログに記録 | 履歴 |
| 失敗 | スナップショットから復元 | ロールバック |

## 7. ライブラリ候補の比較

| 候補 | 役割 | 長所 | 短所 | 判定 |
|---|---|---|---|---|
| Node.js標準`fs`/`path` | 基本操作・パス | 依存が少ない | 高レベル操作を自前実装 | 土台として必須 |
| `fs-extra` | コピー、移動、削除 | 実装が簡潔 | 依存追加 | 採用候補 |
| `chokidar` | 外部変更監視 | OS差を吸収しやすい | 大量イベントの集約が必要 | 採用候補 |
| `globby`/`fast-glob` | ファイル検索 | パターンと除外が容易 | 構成モデルの代替ではない | 補助利用 |
| Electron直接操作 | UIからローカル操作 | 試作が速い | UIと権限が密結合 | 本番非推奨 |
| Gitのみ | 変更・復元 | 差分と復元が強い | 構成意味を理解しない | 復元層として併用 |
| 専用`ProjectOrganizer` | 業務ルール | MQL5依存・コンパイルまで扱える | 初期実装が必要 | 中核として採用 |

## 8. 他設計候補との比較

| 候補 | 利用側の自由度 | 安全性 | 移設性 | 評価 |
|---|---:|---:|---:|---|
| 各ファイルに相対include | 低 | 中 | 高 | 小規模向け。移動に弱い |
| すべて論理include | 中 | 高 | 高 | MQL5探索規則に合わせる必要 |
| 各ファイルに絶対include | 高 | 低 | 低 | 修正箇所が分散 |
| 集約includeを自動生成 | 高 | 高 | 中 | 本用途の推奨 |
| ソース全体を一括書換 | 高 | 中 | 中 | 差分が大きくなりやすい |

## 9. 検討事項

| 論点 | 推奨判断 | 未決定時のリスク |
|---|---|---|
| 絶対include対応 | 最初に最小ファイルでコンパイル検証 | 基盤全体が成立しない |
| プロジェクトルート | 許可ルートを明示設定 | 任意パス操作の脆弱性 |
| 外部変更 | watcherと再スキャンを併用 | モデルと実フォルダの乖離 |
| 同時操作 | プロジェクト単位でロック | 二重移動・モデル破損 |
| 部分失敗 | 一時領域とスナップショット | 中間状態が残る |
| 依存解析 | 初期は明示モデル、後で`.mqh`解析 | 隠れ依存の見落とし |
| PC移設 | 再生成コマンドを用意 | 絶対パスが旧PCを指す |
| 手編集 | 生成ファイルに警告を書く | 次回生成で変更が消える |
| 削除 | ゴミ箱またはスナップショットへ移す | 誤操作で復元不能 |

## 10. 整合性・安全性

| 検証 | 内容 |
|---|---|
| ルート制約 | `resolve(target)`が許可ルート配下か確認 |
| リンク | シンボリックリンクは原則禁止または実体解決後に再検証 |
| パストラバーサル | `..`、UNC、ドライブ変更を拒否または明示許可 |
| 拡張子 | 対象拡張子を限定 |
| 予約領域 | `.git`、生成物、端末管理ファイルを操作禁止 |
| 原子的更新 | 一時ファイル出力後に置換 |
| 監査 | 操作前後のパス、操作者、結果、復元情報を記録 |

## 11. テスト計画

| 層 | 対象 | 例 |
|---|---|---|
| pure | パス・移動計画 | ルート逸脱、衝突、循環 |
| pure | include生成 | 順序、重複、エスケープ、決定性 |
| adapter | `FileSystemAdapter` | 仮想FSまたは一時ディレクトリ |
| integration | transaction | 成功、途中失敗、ロールバック |
| integration | watcher | 外部作成・移動・削除 |
| compiler | MQL5 | 絶対include、連鎖include、生成ファイル |
| parity | 既存構成 | 導入前後のコンパイル結果が同じ |

## 12. 導入段階

| Phase | 内容 | 完了条件 |
|---|---|---|
| 1 | 構成モデル読取とツリー表示 | 既存フォルダを正しく表示 |
| 2 | dry-run移動計画 | 実変更なしで影響範囲を表示 |
| 3 | 移動とロールバック | 失敗時に復元 |
| 4 | 集約include生成 | 最小MQL5ファイルがコンパイル成功 |
| 5 | 実プロジェクト移動 | parity test通過 |
| 6 | watcher・履歴・UI改善 | 外部変更と監査を処理 |

## 13. 最終判断

汎用ファイル操作ライブラリを直接UIへ公開せず、`ProjectOrganizer`という深いModuleを作る。その内部で標準`fs/path`、`fs-extra`、`chokidar`、検索ライブラリ、MetaEditor adapterを使い分ける。

外部Interfaceは次の4つに絞る。

```text
inspect(project)
plan(command)
apply(plan)
restore(changeId)
```

この小さなInterfaceの内部に、パス制約、移動計画、集約include生成、ロールバック、コンパイル検証を閉じ込める。Reactは構成管理の意味だけを扱い、MQL5固有の配置とinclude解決は一箇所に集中させる。
